Overview of Phase 4

After completing the initial setup, implementation, and testing phases, it’s time to evaluate the results. In Phase 4, you will:

  • Review feedback from pupils and staff, assessing usability and performance.
  • Analyse logs and metrics to confirm that the environment meets your technical and educational goals.
  • Decide whether to scale up the solution school-wide or make further adjustments first.
  • Plan for documentation updates, staff training, or additional resources to support a broader rollout.

By the end of this phase, you’ll have a clear picture of the pilot’s success and a roadmap for the future of your classroom computing environment.

Step 1: Feedback Analysis

Review all collected feedback from Phase 3, including pupil surveys, staff observations, and logs of any issues encountered. Determine which areas worked well and which need improvement.

Key Tasks:

  • Read through all pupil feedback notes and identify common themes (e.g., slow logins, difficulty accessing certain files).
  • Discuss findings with teaching and IT staff to confirm the validity and scope of reported issues.
  • Prioritise improvements based on severity and impact on the learning experience.

Reference: Usability.gov - Guidelines on User Feedback

Step 2: Performance and Log Review

Examine system logs, performance metrics, and authentication records to confirm that the technical aspects meet expectations. Check login times, file access speeds, and resource usage.

Key Tasks:

  • Review Samba, Winbind, and system logs (/var/log/samba, /var/log/syslog) for errors or warnings.
  • Check Rclone logs (if enabled) to ensure OneDrive mounts and file accesses occurred without significant errors.
  • Compare actual login times and file access speeds against initial benchmarks noted in earlier phases.
  • Evaluate resource usage (CPU, RAM, network) on the file server and client machines to ensure scalability.

Reference: Samba Logging Documentation

Step 3: Decision Making and Next Steps

With the data gathered and analysed, decide if the pilot environment is ready for a broader rollout. If not, determine what changes are needed before scaling up.

Key Tasks:

  • If all major issues are resolved or manageable, prepare a plan for introducing Linux-based machines into more classrooms.
  • If significant issues remain, document a clear action plan for further refinements before expansion.
  • Discuss the outcome with school leadership and stakeholders to confirm support for the next steps.

Step 4: Documentation and Training

Ensure that staff and pupils have the resources they need to use and support the new environment. Update documentation, create how-to guides, and consider training sessions if necessary.

Key Tasks:

  • Update internal documentation with any changes made during the pilot and testing phases.
  • Provide staff with a basic troubleshooting guide and information on how to escalate issues.
  • Consider short training sessions or video tutorials to help teachers and pupils feel confident using the new system.

Reference: Ansible Documentation Debian Wiki
